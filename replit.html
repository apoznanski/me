Overview

Replit is a multiplayer computer for creating and sharing software. I spent the past summer designing features for Replit's workspace to enhance the developer experience. My work primarily focused on collaborative code commentary and file management.

Redesigning threads

Threads is Replit's in-editor code annotation feature. Threads was first introduced to the workspace in early 2021, and the feature was in need of a revamp to adhere to Replit's new design system and address persistent usability issues. The biggest pain point we needed resolve was overlapping content. Thread indicators and groups were poorly positioned within the workspace and visually blocked code and shell output. We also needed to redesign the feature to adhere to a major workspace design update. This update allowed users to customize their workspace with tabs and panes for any tools—so we needed to convert the threads popover into a pane. Additionally we needed to design a mobile solution for threads, as they were only available on the desktop version of Replit until this point.

Exploration and iterations

We first began exploring different ways that we could address the issue of threads covering content. We experimented with placing threads in the sidebar that hosts the file tree and other workspace tools, and using a chat-like full pane view for conversations. We also experimented with using space within the editor, where users were actually writing code. We felt that the contextual context of the code was important, and that extending the editor to make room for conversations dispersed within the code could make for a more optimized interaction.

We also created a few different translational designs for mobile threads. Designing a solution that allowed the user to view their code and threads at the same time was much more difficult for mobile because of the reduced screen space. We experimented with displaying threads in a bottom drawer on the screen that the user could easily open and close, and a collapsible component placed inline within the code editor.

Mobile threads exploration.

After a few rounds of critique and some more experimentation with prototyping, it became clear that the inline threads provided the most ergonomic interaction. Placing threads in an inline component that can expand and collapse alleviated the issue of covering content. Having individual threads within the editor also made the interaction model easier to define. We decided to keep individual or "singleton" threads inline in the editor, and have a global threads "inbox" pane which allows the user to view cards of all the threads in their files. All of the conversation interactions like starting and replying to threads occur within the editor. Within the threads inbox, we added the ability to search, sort, and filter all the threads within a project. Based on that interaction model, we defined the basic interface architecture below.

We designed multiple iterations of inline threads, changing different aspects of the design like the thread indicator, the header of the inline component, and the elevation level (as indicated by the color).

Final solutions

We decided to move forward with a thread indicator in the right gutter of the editor, so as not to be confused or interfere with breakpoints on the line numbers. We also chose to use a lower elevation color for the thread component, to give the impression of peeking inside the code, or opening a drawer within the editor. We wanted to be very intentional about the design of the inline component, because we felt strongly about using it as a primitive for other in-editor AI features like Replit's "explain code."

Inline thread final solution.
Video demo of creating a thread.

For the threads pane we created a global inbox that allows the user to search, sort, and filter all the threads in their project. The inbox contains an accordion for each file within the project that can be opened to view all the threads in that specific file. Threads are displayed as cards which show the line number and a snippet of the first line of code that the thread refers to, and the inital comment from the user who started the thread. Clicking on a thread card from the pane opens the thread in the editor.

We used the same interface components and interaction model on desktop and mobile. Placing threads inline allowed us to maximize the small screen space we were working with, and prevented users from having to context switch between code and comments constantly.

Conclusion

The threads redesign has now been fully shipped on Replit! We've received overwhelmingly positive feedback, and I'm extremely proud to have been a part of this project. The inline component designed for threads will also become a new workspace primitive, allowing users to harness the power of Replit's AI features without ever having to leave the code editor. Working on this feature taught me a lot about fighting my fastidiousness. Replit has a reputation for shipping fast, and I soon realized that perfectionism hinders progress. Sharing work consistently, getting feedback often, and iterating quickly pushes a design much further than meticulous overthinking. I had to get over my fear of sharing work that I wasn't 100% happy with—the worst thing someone can say is that they know you can do better, which is really motivating! Each and every person on the Replit design team has had an influence on my growth as a designer. I'm grateful to have learned from the best, and contributed to such an amazing product pioneering the future of collaborative computing!
